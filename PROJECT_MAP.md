# Project Map: ProVision-FedWatch

## High-Level Architecture

The ProVision-FedWatch system is a Node.js application designed to monitor sam.gov for federal contracting opportunities, score them for relevance against multiple company profiles, and send alerts to Slack.

The system is composed of the following key modules:

*   **`bot/`**: The core application logic.
    *   `runner.js`: Main process loop that iterates through company profiles.
    *   `sam_client.js`: Client for interacting with the SAM.gov API.
    *   `enrich.js`: Downloads and extracts text from PDF attachments.
    *   `ai.js`: Interfaces with the Gemini AI for scoring and analysis.
    *   `scoring.js`: Calculates a deterministic score and prepares data for the AI.
    *   `storage.js`: Manages the local SQLite database for opportunities.
    *   `slack.js`: Sends formatted notifications to Slack.
    *   `cli.js`: Provides a command-line interface for running the bot.
*   **`config/`**: Configuration files.
    *   `opportunity-bot.json`: Defines company profiles, filters (NAICS codes, keywords), and Slack integration details.
*   **`worker/`**: Scripts for running the bot as a scheduled process.
*   **`supabase/`**: Supabase functions (currently used for testing and dispatching).
*   **`.github/workflows/`**: GitHub Actions for CI and scheduled bot runs.

## Multi-Profile System

The bot was refactored to support multiple company profiles from a single configuration file (`config/opportunity-bot.json`). The `bot/runner.js` module iterates through each defined profile and executes the fetching, scoring, and alerting pipeline independently. This allows the system to serve different business entities (e.g., "RMC Integration Services LLC" and "Provisions Unlimited") simultaneously.

## Scheduling and Automation

The bot is designed to be run on a schedule. This is currently implemented using a GitHub Actions workflow (`.github/workflows/opportunity-bot-scheduled-run.yml`) which runs twice daily. The workflow handles environment setup, dependency installation, and execution of the bot, with status notifications sent to a dedicated Slack channel.

## Core Logic Summary

This section summarizes the core logic of the bot, extracted from the source code to ensure it is preserved.

### Fit-Rating and Scoring Logic (`bot/scoring.js`)

The primary scoring is determined by the `deterministicScore` function, which works as follows:

1.  **Initial Score**: Starts with a base score of 50.
2.  **Keyword Matching**: It searches through a combined text block containing the opportunity's title, description, agency, NAICS code, and other fields.
    *   **Positive Keywords**: For each match from a predefined list of positive terms (e.g., "IT managed services", "telecommunications"), the score is increased by 5.
    *   **Negative Keywords**: For each match from a list of negative terms (e.g., "construction", "roofing"), the score is decreased by 10.
3.  **Final Score**: The score is capped between 0 and 100.
4.  **Labeling**: The final score is converted to a label ("GOOD_FIT", "MAYBE", "NOT_A_FIT") based on configurable thresholds.

*Note: While specific NAICS codes like 541511 are included in the text search, they are not assigned any special weight; the logic is primarily keyword-driven.*

### Fatal Error Handling (`bot/cli.js`)

The main application entry point (`bot/cli.js`) includes a top-level `catch` block to handle any uncaught exceptions during the bot's execution. This ensures that any fatal error is logged to the console in a standardized format before the process exits.

The implementation is:
```javascript
main().catch((error) => {
  console.error("[opportunity-bot] Fatal error:", error.message);
  process.exit(1);
});
```

### Slack Notification Formatting (`bot/slack.js`)

The `buildSlackPayload` function constructs a rich notification using Slack's Block Kit API. The final, stable format is modular and includes the following blocks in order:

1.  **Header**: Displays the opportunity title and a Fit-Rating label (e.g., "[GOOD FIT ✅]").
2.  **AI Summary**: A section containing the plain-English summary of the opportunity generated by the AI.
3.  **Divider**: A visual separator.
4.  **Key Fields**: A section of two-column fields displaying core data like Agency, Solicitation #, Due Date, and NAICS code.
5.  **AI Analysis Sections**: Separate, optional sections are appended for:
    *   Required Skillsets
    *   Top Reasons (for the fit rating)
    *   Key Risks
    *   Must Check Items (directing the team to specific document sections)
6.  **Links**: A section containing a list of relevant hyperlinks (e.g., to SAM.gov, attachments).
7.  **Context**: A final, smaller context block displaying the numerical Fit Score.

## Versioning Strategy

To ensure we can always return to a stable, working state, this project uses a `git tag` based versioning strategy.

*   **`functional-version`**: This tag marks the commit representing the last known "good" state of the bot.
*   **Updating**: The tag can be updated to a new stable point upon request.
*   **Reverting**: The entire project can be reverted to this stable point by running `git checkout functional-version`.
